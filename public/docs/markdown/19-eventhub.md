EventHub is the package registry for Volnux. It's where teams publish events, workflows, and agents. It's where other teams discover and consume them. It's the enforcement point for the governance contract — every package has a validated manifest, every dependency is resolved, every deprecation is tracked.

## The EventHub Registry
The EventHub registry is the central repository for Volnux packages. It stores packages, validates manifests, resolves dependencies, and serves packages to consumers.

Core capabilities:
* **Package storage**: Event packages, workflow packages, and agent packages with versioned releases.
* **Manifest validation**: Every package is validated against the JSON Schema at publish time. Invalid packages are rejected before they reach consumers.
* **Dependency resolution**: The registry resolves transitive dependencies. If a workflow depends on `trading-events>=2.1.0` and `trading-events` depends on `market-data-lib>=3.0`, the registry ensures all dependencies are available and compatible.
* **Version management**: Semantic versioning enforced. Breaking changes require major version bumps. The registry prevents removal of events before their declared `removal_version`.
* **Search and discovery**: Tags, categories, keywords. Teams can find packages by name, description, event type, or workflow pattern.

Public and private registries:
* **Public EventHub**: `https://eventhub.volnux.dev` — Community packages, open-source events, reference workflows.
* **Private EventHub**: Self-hosted or Volnux Cloud — Enterprise packages, proprietary events, internal workflows. Organisation-scoped access control.

## Publishing Packages
Publishing is a three-step process: generate the manifest, validate it, push to the registry.

### Event Packages
An event package contains reusable `EventBase` subclasses. It's a library of building blocks.

Project structure:
```text
trading-events/
├── trading/
│   ├── __init__.py
│   ├── event.py              # class CalculateRisk(EventBase)
│   ├── other_event.py        # class ValidateTrade(EventBase)
│   └── another_event.py      # class FetchMarketData(EventBase)
└── volnux.manifest.json      # Generated by CLI
```

Publishing:
```bash
cd trading-events/
volnux manifest generate       # Detect project type, generate manifest
volnux manifest validate       # Validate against JSON Schema
volnux publish                 # Push to EventHub
```

The CLI detects the project type (event project — no `workflows/` directory, no `WorkflowConfig`, just event classes in namespace directories). It introspects each `EventBase` subclass, extracts class names, module paths, versions, and `INIT_PARAMS_SCHEMA`, and generates the manifest.

What the registry receives:
```json
{
    "package_type": "event",
    "package": {
        "name": "trading-events",
        "version": "2.1.0"
    },
    "events": [
        {
            "class": "CalculateRisk",
            "module": "trading.event",
            "name": "calculate_risk",
            "version": "1.3.0",
            "categories": ["ANALYZE"],
            "init_params": {
                "confidence": {"type": "float", "required": false, "default": 0.95}
            }
        }
    ]
}
```

### Workflow Packages
A workflow package contains a complete, runnable workflow — Pointy-Lang definition, pipeline, events, triggers, and configuration.

Project structure:
```text
trade-reconciliation/
├── config.py
├── init.py
├── workflows/
│   └── trading/
│       ├── workflow.py       # class TradingConfig(WorkflowConfig)
│       ├── pipeline.py       # class TradingPipeline(Pipeline)
│       ├── events.py         # Local event definitions
│       └── trading.ptl       # Pointy-Lang source
└── volnux.manifest.json      # Generated by CLI
```

Publishing:
```bash
cd trade-reconciliation/
volnux manifest generate       # Detect project type, parse ready(), extract graph
volnux manifest validate
volnux publish
```

The CLI detects the workflow project type (`workflows/` directory with `WorkflowConfig` subclasses). It introspects `ready()` to extract registered triggers and sources. It reads the `.pty` file. It generates a manifest with the complete workflow specification.

### Agent Packages
An agent package contains a governed AI agent — `AgentEventBase` subclass with declared tools, handoff permissions, and prompt version.

Publishing:
Agent packages follow the same flow as event packages. The manifest includes agent-specific metadata:

```json
{
    "package_type": "event",
    "package": {
        "name": "trading-agents",
        "version": "1.0.0"
    },
    "events": [
        {
            "class": "TradingAnalyst",
            "module": "trading_agents.analyst",
            "version": "1.0.0",
            "categories": ["AI", "AGENT"],
            "agent": {
                "entry_point": "TradingAnalyst",
                "llm_providers": ["anthropic"],
                "tools": ["FetchMarketData", "CalculateRisk", "ExecuteTrade"],
                "permitted_handoffs": ["ComplianceReviewer", "SeniorTrader"],
                "prompt_version": "1.2.0",
                "max_reasoning_steps": 15,
                "token_budget": 100000
            }
        }
    ]
}
```

The agent declaration is machine-validated. A consumer can see exactly which LLM providers the agent uses, which tools it can call, and which other agents it can hand off to — before the agent ever runs.

## Consuming Packages
Workflows consume packages via `WorkflowSource` declarations in `ready()`.

### WorkflowSource
```python
from volnux import WorkflowSource, RegistrySource, SourceCredentials

def ready(self):
    self.register_source(WorkflowSource(
        name="trading-events",
        source_type=RegistrySource.HUB,
        location="trading-team/trading-events",
        version="2.1.0",
        events=["CalculateRisk", "ValidateTrade"],  # Selective loading
        credentials=SourceCredentials(
            api_key=os.environ["EVENTHUB_API_KEY"]
        ),
    ))
```

Source types:

| Type | Description | Example |
| :--- | :--- | :--- |
| `HUB` | EventHub registry | `location="team/package"` |
| `PYPI` | PyPI package | `location="volnux-trading-events"` |
| `GITHUB` | GitHub repository | `location="company/volnux-events"` |
| `LOCAL` | Local filesystem | `location="./local-events"` |

### Package Loader (PyPI, GitHub, Hub)
At initialisation time, Volnux resolves all sources:

```python
class PackageLoader:
    async def load_from_source(self, source: WorkflowSource):
        if source.source_type == RegistrySource.HUB:
            return await self._load_from_hub(source)
        elif source.source_type == RegistrySource.PYPI:
            return await self._load_from_pypi(source)
        elif source.source_type == RegistrySource.GITHUB:
            return await self._load_from_github(source)
        elif source.source_type == RegistrySource.LOCAL:
            return await self._load_from_local(source)
```

**Hub loader:**
* Authenticate with EventHub API using credentials.
* Request package metadata for the specified version.
* Download the package (wheel or source distribution).
* Validate the manifest signature.
* Install into the runtime environment.
* Import requested event classes.
* Apply manifest metadata (version stamps, deprecation warnings).

**PyPI loader:**
* `pip install package==version` into the runtime environment.
* Import requested event classes.
* Validate against the package's embedded manifest.

**GitHub loader:**
* Clone or fetch the repository at the specified tag or commit.
* Install from the local path.
* Import and validate.

**Local loader:**
* Import from the local filesystem path.
* No network access required.

## Dependency Resolution
The registry resolves transitive dependencies automatically:

```text
Workflow: trade-reconciliation@2.1.0
    ├── trading-events@2.1.0
    │   ├── market-data-lib@3.0.1
    │   └── risk-engine@1.5.0
    └── notification-events@1.5.0
        └── slack-sdk@3.0.0
```

Conflict detection:
If `trading-events` requires `market-data-lib>=3.0,<4.0` and `notification-events` requires `market-data-lib>=2.0,<3.0`, the resolver detects the conflict at initialization time. The workflow fails to load with a clear error:

```text
Dependency conflict:
  trading-events@2.1.0 requires market-data-lib>=3.0,<4.0
  notification-events@1.5.0 requires market-data-lib>=2.0,<3.0
Resolution: No compatible version found.
```

The error happens at load time, not mid-execution. The operator knows about the conflict before any events run.

## Zero-Code Workflow Execution
The ultimate expression of composability: run a workflow without writing any code.

A consumer creates a single file:

```python
# trading_workflow.py
from volnux.engine.workflows import WorkflowConfig
from volnux import WorkflowSource, RegistrySource

class TradingWorkflow(WorkflowConfig):
    name = "trading"
    version = "1.0.0"
    mode = "CFG"
    
    def ready(self):
        self.register_source(WorkflowSource(
            name="trade-reconciliation",
            source_type=RegistrySource.HUB,
            location="compliance-team/trade-reconciliation",
            version="2.1.0",
        ))
```

```bash
volnux workflow run trading
```

What happens:
* The CLI discovers the `TradingWorkflow` config.
* The engine calls `ready()` — registers the EventHub source.
* The package loader downloads `trade-reconciliation@2.1.0` from EventHub.
* The manifest is read. The Pointy-Lang file is loaded. The pipeline is discovered. The triggers are registered.
* The engine compiles the graph and executes the workflow.
* The consumer never wrote an event. Never wrote Pointy-Lang. Never defined a pipeline. They pointed at a registry. The workflow ran.

This is "composable without chaos" at the workflow level. A compliance officer can run a published workflow without understanding its internals. A team can share complete workflows through EventHub. Governance is preserved — the version is pinned, the manifest is validated, the audit trail records exactly what ran and where it came from.
